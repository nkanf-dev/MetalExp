# 2026-04-11 世界可游玩里程碑设计

## 摘要

这份设计文档定义 `MetalExp` 的下一个核心里程碑：让原生 Metal backend 从“菜单和 atlas 可用”推进到“能够进入世界并短时间基本游玩”。

这个里程碑的重点不是一次性补齐所有渲染功能，也不是提前转向完整 shader/toolchain 通用化，而是围绕世界主路径建立一个严格、可诊断、可逐步扩展的垂直切片。

项目在这个阶段的产品目标是：

- 能进入单人世界
- 能维持几分钟基本游玩
- 允许部分特效、天气、粒子、后处理或特殊材质暂时不完整
- 不允许主画面长期紫屏、整屏黑屏、窗口卡死、持续闪烁、或因为关键 pass 缺失而完全不可辨认

这份设计同时明确一个执行原则：

- 优先实现世界主路径必需的 pass 和状态语义
- 对少数非关键、暂未支持的 pass，直接明确报错并输出上下文
- 不接受悄悄回退成大面积 no-op 或难以定位的“似乎哪里不对”

---

## 为什么现在进入这个里程碑

截至目前，项目已经跨过了“完全不能工作”的早期 bring-up 阶段，完成了几项关键基础工作：

- Metal backend 能启动并接管基础表面生命周期
- GUI 主路径已经恢复，主菜单不再因为 atlas 空白而大面积透明
- `animate_sprite_blit` 已经从假路径恢复到严格 native Metal 路径
- command submission 模型已经从细粒度立即提交推进到可批量提交
- 紫屏卡死问题已经定位并修正到跨 `CommandEncoder` 的共享 command context 语义

这说明项目已经不再停留在“证明 Metal backend 能创建对象”的阶段，而是进入了“需要让真实 Minecraft 渲染主路径连续运行”的阶段。

如果继续只围绕菜单、atlas、单个 GUI pipeline 打补丁，收益会迅速下降。下一步最有价值的目标，是用一个更接近真实使用场景的里程碑来拉动架构前进：进入世界，并且能短时间游玩。

---

## 里程碑目标

### 成功定义

本里程碑完成时，Metal backend 应满足以下条件：

- 玩家可以从主菜单进入单人世界
- 进入世界后可以连续游玩几分钟，而不需要切换回其他 backend
- 世界主画面在视觉上基本可辨认，地形与主要几何不会因为深度完全失效而严重互相穿插
- HUD、物品栏、暂停菜单、基础 GUI 继续保持可用
- 如果运行撞到未支持但非关键的渲染路径，系统能够给出明确、结构化、可定位的失败信息

### 不属于本里程碑完成线的内容

以下内容可以暂时不完整，只要它们不阻断“进入世界并短时游玩”的主目标：

- 粒子、天气、云层、天空、后处理
- 少数特殊实体材质或非常规 pipeline
- 完整 shader/toolchain 通用化
- 全量渲染特性对齐 OpenGL/Vulkan
- 广泛 renderer mod 兼容
- 所有性能问题一次性解决

---

## 非目标

本阶段不做以下方向：

- 不把目标改成“所有世界相关 pipeline 都必须完整支持”
- 不用大量 silent fallback 掩盖未支持路径
- 不为了短期进世界而把主路径再次拉回 CPU 模拟渲染
- 不跳过世界主路径语义，直接把大量 pipeline 名字继续堆进硬编码特判
- 不把本里程碑扩展成完整 shader 编译和反射系统的总实现

---

## 当前约束与现实

当前代码基础决定了这个里程碑必须正视两个事实。

### 1. 目前已经有一部分可用 native path

`MetalRenderPassBackend` 已经能处理一组受控的 GUI/panorama/atlas pass，并且能通过 native bridge 在 Metal 侧执行：

- GUI color
- GUI textured
- GUI opaque textured background
- GUI premultiplied alpha textured
- panorama
- animate sprite blit

这说明“Java 侧 thin hook + native 侧执行”的总体思路是成立的。

### 2. 当前 render pass 语义仍然明显偏向菜单阶段

现有实现还没有为世界主路径补齐最低限度的通用图形语义，尤其包括：

- 颜色目标假设偏窄，当前实现基本围绕 `RGBA8_UNORM`
- depth attachment 语义没有真正打通
- `clearDepthTexture()` 仍是空实现
- 世界 pass 的全貌还没有结构化 census
- 不支持 pipeline 的报错仍偏“单点异常”，不够像一个可迭代的诊断系统

因此，本里程碑不能简单等同于“再加几个 pipeline 名字”，而必须把主路径所需的最低 render pass 语义一并补上。

---

## 推荐方案

推荐采用“世界可游玩垂直切片”方案，而不是继续扩展菜单阶段的零散特判。

其核心思路是：

1. 先把世界实际经过的 pass 和状态面完整暴露出来
2. 再补足主路径必需的最低图形语义
3. 之后按主路径能力族来补支持，而不是按零散 pipeline 名字打补丁
4. 对非关键且未支持的部分，明确 fail-fast，并带上能驱动下一轮修复的上下文

这条路线的优势是：

- 与“可基本游玩几分钟”的目标直接对齐
- 每一轮推进都有明确的阻塞点
- 不会把代码继续推向越来越难维护的大量特判
- 能让世界渲染问题逐步沉淀成有边界的实现和诊断模型

---

## 技术设计

### 一、建立世界 pass census 与结构化失败信息

第一步不是立即补世界 pipeline，而是先让系统能够准确描述“世界阶段究竟跑到了什么”。

需要增加一套结构化的 pass 观察与失败信息，至少覆盖以下维度：

- pipeline id 或 location
- 颜色目标格式、尺寸、mip 信息
- 是否存在 depth target，以及 depth format
- draw 类型：`draw` / `drawIndexed` / `drawMultipleIndexed`
- vertex format、index type
- 绑定纹理名、uniform 名
- 是否走 native path
- 当前 command context 和 surface 语义是否共享

当遇到未支持路径时，异常和日志不应只是：

- “某某 pipeline 不支持”

而应尽量携带：

- 是哪个 pass
- 缺的是 pipeline 支持，还是颜色目标格式，还是 depth 语义，还是绑定资源条件
- 这条 pass 是否可能属于世界主路径阻塞项

这会成为整个里程碑的导航系统。没有这一层，后续所有“进世界”调试都会重新退化成盲试和日志猜测。

### 二、补齐世界主路径所需的最低 render pass 语义

世界可游玩首先依赖一组最低语义成立，而不只是“能调用 draw”。

本阶段需要优先补齐：

- depth attachment 的真实传递和校验
- depth clear 行为
- depth compare / depth write 的最低支持
- render target 校验从“菜单阶段假设”扩展到世界主路径实际需要
- 保证 scissor、基础 blend、纹理采样、命令提交流程在世界阶段仍与当前 batching 语义兼容

这里的目标不是立即做出完整通用 render graph，而是保证世界主通道的基础几何关系成立。

如果 depth 仍然是空壳，即使某些世界 pipeline 被“跑起来了”，画面也可能因为前后顺序彻底错误而变得不可游玩。

### 三、按主路径能力族补 pipeline，而不是按零散名字补洞

当 pass census 和基础语义具备后，下一步应按能力族推进，而不是按“又报了一个新 pipeline 名字”来修。

推荐优先顺序如下：

1. 世界不透明 textured/color geometry
2. cutout / alpha-test 风格几何
3. 基础实体、手持物品、HUD 仍会经过的可复用纹理几何路径
4. 非关键的调试或边缘 geometry

这样做的原因是，真正决定“能否短时间游玩”的，通常不是某个单独 pipeline 常量，而是一类相近的状态组合是否成立。

能力族推进也更容易逼近未来的 shader/toolchain 通用化，因为它鼓励我们围绕语义边界组织代码，而不是围绕一长串特殊名字分支。

### 四、对非关键未支持路径统一 fail-fast

本里程碑明确拒绝“大面积静默降级”。

对于当前仍不属于主路径的 pass，例如：

- 粒子
- 天气
- 某些后处理
- 少数特殊实体材质
- 边缘效果 pipeline

如果尚未支持，应统一走明确的失败机制：

- 抛出包含上下文的异常，或
- 在严格模式中终止并输出诊断快照

这个失败机制必须是“对开发友好”的，而不是“对用户模糊”的。

这样做的价值在于：

- 每一轮新的世界渲染失败都能形成具体工作项
- 不会因为 no-op 把主路径问题掩盖成更难定位的视觉异常
- 可以更自然地把未来的支持计划写成按能力族推进的 docs/plans 文档

## 代码层落点

### `MetalRenderPassBackend`

这里会是本里程碑最核心的 Java 侧收口点，负责：

- pass census
- pipeline 分类
- render pass 状态校验
- depth/target 条件校验
- 未支持路径的结构化 fail-fast

它仍应保持“Minecraft hook 薄、策略集中”的方向，但要从“菜单阶段专用适配器”演进成“世界主路径渲染诊断与调度收口点”。

### `MetalCommandEncoderBackend`

继续承担 command batching 的 Java 侧入口，并在需要时承载：

- 与 render pass 生命周期更一致的提交行为
- 世界阶段额外诊断信息的挂接
- depth / surface 相关提交时序的验证

### `MetalDeviceBackend` 与相关资源类

需要补足最低限度的设备/资源语义一致性，避免继续停留在“bootstrap backend”假设：

- render target 与 depth target 的使用边界更清晰
- 资源格式和支持范围的报错更明确
- 世界主路径需要的纹理/缓冲约束能够提前暴露

### native bridge

native 侧仍应优先承载性能敏感路径，但只在主路径真实需要时新增入口。

本里程碑不鼓励因为“可能以后用得上”就提前扩张大量 native API。原则仍然是：

- Java 侧判定和调度尽量薄
- 真正性能敏感且语义已稳定的路径尽量 native
- 不稳定、仍在摸索阶段的策略判断先留在 Java 侧收口

## 验收标准

### 必须达成

- 可以从主菜单进入单人世界
- 进入世界后可连续游玩数分钟
- 世界主画面基本可辨认
- 主要不透明几何的深度关系基本正确
- HUD、物品栏、暂停菜单保持可用
- 不出现长期紫屏、整屏黑屏、窗口卡死、持续闪烁
- 对未支持 pass 的失败信息具备明确 pipeline 与语义上下文

### 可以暂时接受

- 粒子、天气、天空、后处理不完整
- 少量特殊实体或边缘材质报错
- 部分半透明效果不正确
- 性能仍需继续优化，但不至于阻断短时间游玩

---

## 推荐执行顺序

### 阶段 1：世界 pass census 与失败信息基础设施

输出物：

- 世界阶段 pass 结构化日志
- 未支持路径的统一异常模型
- 第一轮世界阻塞项清单

### 阶段 2：最低限度的 depth / render pass 语义补齐

输出物：

- depth target 真正接入
- depth clear 与最基本比较/写入语义
- 与世界主路径一致的 render pass 校验

### 阶段 3：按主路径能力族补支持

输出物：

- 不透明世界几何可出图
- 基础 cutout / alpha-test 路径逐步接通
- HUD / 手持物品 / 常见纹理几何与世界阶段协同工作

### 阶段 4：收口未支持边界与文档

输出物：

- 明确列出当前仍未支持的非关键 pass 范围
- 形成下一轮迭代的具体阻塞点
- 用日志与计划文档固定当前世界可玩边界

## 风险与取舍

### 风险 1：世界主路径远比菜单路径复杂

这意味着本里程碑很可能不会在第一轮就把“进入世界”一次性打通，而是会经历多轮：

- census
- 阻塞项识别
- 能力族补齐
- 再次进入世界验证

这是正常路径，不应被误判为设计失败。

### 风险 2：depth 语义可能牵动 native 与 Java 两边

depth 不是单点补丁，可能会同时牵涉：

- Java 侧 render pass 建模
- native 侧 pipeline / render pass 编码
- 资源格式和 attachment 条件约束

因此必须把它当成本里程碑的一等主题，而不是附带小修。

### 风险 3：继续堆特判会带来中期维护爆炸

如果在世界阶段仍然沿用“发现一个 pipeline 就补一个分支”的策略，短期也许能推进一点，但后续会更难转向通用化。

所以本设计明确要求：

- 以能力族而不是零散 pipeline 名字为组织方式
- 让诊断系统先于补洞扩张

---

## 后续衔接

当这个里程碑完成后，项目将处在一个更有价值的位置：

- Metal backend 不再只是在菜单阶段可演示
- 世界主路径已有一个真实可运行的垂直切片
- 下一阶段可以更理性地决定是继续补世界渲染覆盖面，还是转向更通用的 shader/toolchain 体系

也就是说，这个里程碑不是临时修 world 的一次战术补丁，而是整个项目从“证明能跑”走向“形成真实 backend 语义”的关键转折点。
