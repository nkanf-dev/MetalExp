# Minecraft 26.2 Vulkan 与 Metal 路线研究报告（MetalExp 项目整合版）

## 1. 目的

本报告是 `MetalExp` 项目的项目内研究基线文档，用来回答两个问题：

1. Minecraft `26.2-snapshot-1` 的现代 Blaze3D/Vulkan 架构到底给了我们哪些接入点
2. 基于 Fabric Loader、Mixin 和原生桥接，`MetalExp` 这种原生 Metal backend 项目是否值得立项，以及应当怎么落地

这份文档不是对上游研究材料的逐字复制，而是结合 `MetalExp` 项目目标做过一次项目化整合。

相关上游研究文档仍然保留在全局文档区：

- `/Users/nkanf/docs/reverse/2026-04-09-minecraft-26.2-vulkan-research-report_cn.md`
- `/Users/nkanf/docs/reverse/2026-04-09-minecraft-26.2-vulkan-research-report.md`

## 2. 项目结论先行

先给最重要的结论：

1. Minecraft `26.2-snapshot-1` 中的 Vulkan backend 是真实可工作的现代 backend，不是实验残骸。
2. Blaze3D 已经被重构成显式 backend-oriented 架构，这为原生 Metal backend 提供了第一次现实可行的技术边界。
3. `MetalExp` 可以借助 Fabric Loader 和 Mixin 接管设置项与启动期 backend negotiation，但不能把“渲染器本体”误判为只是几个 mixin。
4. 作为大项目，`MetalExp` 的本质是“第三方 Blaze3D backend program”，而不是传统功能模组。
5. `MetalExp` 的最可信首期 shader 路线是 `GLSL -> SPIR-V -> MSL`。
6. 真正的高成本主要集中在：
   - shader 编译与资源绑定语义
   - command encoding 与同步语义适配
   - snapshot 漂移带来的适配维护

## 3. Mojang 当前架构给了什么

### 3.1 已形成的 backend seam

研究显示，现代 Blaze3D 里至少已经形成以下关键边界：

- `GpuBackend`
- `GpuDeviceBackend`
- `GpuSurfaceBackend`
- `CommandEncoderBackend`
- `RenderPassBackend`

这意味着对于 `MetalExp` 来说，目标不再是“替换整个 Minecraft 渲染器”，而是：

- 在 backend negotiation 处把 `MetalBackend` 接进去
- 在 backend 抽象下实现完整的设备、surface、shader、pipeline 和命令提交流程

这是项目成立的根本前提。

### 3.2 Vulkan 已经是现代 explicit backend

在 `26.2-snapshot-1` 的 Vulkan 实现里，已经观察到：

- Vulkan 1.2 能力门槛
- `VK_KHR_dynamic_rendering`
- `VK_KHR_synchronization2`
- `VK_KHR_push_descriptor`
- VMA 内存分配
- `shaderc` 驱动的 GLSL 到 SPIR-V 编译
- surface acquire / blit / present 收口路径

这说明 Mojang 当前不是用最保守的 Vulkan 最小子集，而是在按现代 explicit API 的心智模型做 backend。

对 `MetalExp` 的意义是：

- 好消息：高层抽象更现代，`RenderPassBackend` 和 command encoder 的概念更接近 Metal
- 坏消息：同步、资源绑定、shader pipeline 的语义映射也更硬核

## 4. 为什么原生 Metal 现在才开始变得现实

旧时代 OpenGL 中心化的 Minecraft 渲染架构，不适合在 mod 里引入真正的 native Metal backend，因为几乎没有清晰的 backend 分割线。

在 `26.2-snapshot-1` 这一代：

- backend 选择点是明确的
- 设备和 surface 生命周期是后端化的
- shader 和 pipeline 已经是显式资源模型
- 输出路径也不再死绑传统 OpenGL swapchain 思维

这让 `MetalExp` 首次具备了现实工程基础。

注意，这里“现实”不等于“便宜”。只是从“几乎不适合立项”提升到了“高成本但合理”。

## 5. Fabric Loader + Mixin 能做到什么，做不到什么

### 5.1 能做到的部分

借助 Fabric Loader 和 Mixin，`MetalExp` 可以可靠接管：

- 图形 API 设置项 UI
- 配置读写入口
- Minecraft 启动期 backend negotiation
- 某些必要的 bootstrap 生命周期
- 失败回退和诊断注入

这足够把一个第三方 backend 挂到现代 Blaze3D 的入口边界上。

### 5.2 做不到替代的部分

Fabric Loader 和 Mixin 不能替代以下系统：

- native Metal bridge
- shader toolchain
- 资源绑定布局
- Metal command encoding
- pipeline state 生命周期
- surface/drawable/present 生命周期

所以 `MetalExp` 不能被理解为：

- “通过 mixin 把 Minecraft 改成 Metal”

更准确的理解是：

- “通过 mixin 接管入口，再用一个完整 backend 实现把 Metal 接进去”

## 6. 为什么不直接给 Mojang enum 动态加 `METAL`

项目讨论里已经明确排除了“动态扩展 `PreferredGraphicsApi` enum”方案。原因包括：

- Java enum 动态扩常量本身脆弱
- Mojang 的设置项还和 codec/序列化绑定
- `values()`、switch 表、选项构造和后续 snapshot 漂移都会放大维护风险

因此 `MetalExp` 的产品化路线是：

- 不扩展原版 enum
- 覆盖原版设置项
- 提供自己的 `Metal / Vulkan / OpenGL` 选择
- 真实配置由 `MetalExp` 自己持有
- 启动时根据 `MetalExp` 配置生成 backend 选择顺序

这条路比“enum 注入”更像一个能长期维护的工程项目。

## 7. MetalExp 的推荐接入方式

### 7.1 设置项覆盖

项目应当直接替换原版 graphics API 选项，而不是隐藏含义或挟持 `VULKAN` 展示文案。

理由：

- 用户心智清晰
- 支持文档清晰
- 未来扩展 fallback、strict、capability gating 时更自然
- 不会引入“界面显示 Vulkan，实际运行 Metal”的歧义

### 7.2 启动期 backend negotiation 接管

项目应当在 Minecraft 启动时接管 backend negotiation，而不是把大量 mixin 撒到渲染期。

设计目标：

- `Metal` 选项时优先 `MetalBackend`
- 可选回退到 `Vulkan`、`OpenGL`
- `strict` 模式下则直接失败
- `Vulkan/OpenGL` 选项尽量走原版路径

### 7.3 保持 Window 宿主模型

项目不应重做 Minecraft 的窗口系统，而应：

- 保留 GLFW / Window 宿主
- 从原生桥接层把宿主窗口解包成 Cocoa window/view
- 在其上附着 `CAMetalLayer`
- 由 `MetalGpuSurface` 管理 drawable 和 present 生命周期

## 8. 项目推荐架构

### 8.1 典型 Fabric mod 外壳 + 原生子系统

项目结构推荐保持“典型 Fabric mod 项目格式”，同时加入长期 renderer 工程所需子系统：

- `src/main`
- `src/client`
- `shared/`
- `native/`
- `docs/`

这是为了同时满足两件事：

- Minecraft/Fabric 适配层保持标准、可理解
- renderer 本体与 native bridge 不被 Fabric 结构拖成一团

### 8.2 五层逻辑模块

推荐的五层模块是：

1. bootstrap 层
2. shared model 层
3. Java backend 层
4. shader/toolchain 层
5. native bridge 层

这不是为了“整齐”，而是为了压住未来的复杂度。

## 9. Shader 路线判断

### 9.1 为什么选 `GLSL -> SPIR-V -> MSL`

`MetalExp` 首期推荐路线是：

`GLSL -> SPIR-V -> MSL`

理由：

- 与 Mojang 当前 Vulkan-era shader 输入语义最接近
- 可以复用已有 preprocessing 思路
- SPIR-V 可以作为稳定 IR
- 更适合先做出真正工作的 backend

### 9.2 最大难点不只是转成 MSL

很多人容易低估的地方在于：

- 真正的难点不是“能不能生成 MSL”
- 而是“如何稳定地把高层资源绑定语义映射成 Metal 绑定模型”

所以 `MetalExp` 需要一个 engine-owned 的中间资源布局模型，而不是长期沿用 Vulkan descriptor 思维。

第一版推荐：

- 使用 engine-managed slot map
- 不上 argument buffers
- 优先保证可调试和可诊断

## 10. Command / Sync / Present 路线判断

### 10.1 运行时模型应当保守

第一版应当采用保守策略：

- 每帧一个主 command buffer
- render/blit 顺序明确
- drawable present 在帧末尾统一调度
- correctness first

实验性公开项目不该一开始就追求花哨的并行提交模型。

### 10.2 Dynamic rendering 对 Metal 是利好

研究里已经确认 Mojang 当前 Vulkan 路线偏向 dynamic rendering 思维，而非传统重 render pass 模型。

这对 `MetalExp` 是明确利好，因为：

- `RenderPassBackend` 更容易映射到 `MTLRenderCommandEncoder`
- pass 构建模型与 Metal 更接近
- 减少某些老式 Vulkan 兼容心智带来的负担

### 10.3 Surface 输出应当后置收口

项目应尽量保持：

- 主渲染到内部 render target
- 最后再输出到 presentable drawable

这样做的好处是：

- 降低场景渲染和 surface 的耦合
- 对 resize、调试、capture 更友好
- 与 Mojang 当前 Vulkan 输出路径方向一致

## 11. Native bridge 判断

### 11.1 原生桥接不是可选项

如果目标是原生 Metal backend，就必须接受原生桥接层成为正式项目组成部分。

它至少要负责：

- Cocoa window/view 解包
- `CAMetalLayer`
- `MTLDevice`
- `MTLCommandQueue`
- `MTLLibrary`
- `MTLRenderPipelineState`
- drawable/present

### 11.2 Java 不应承担过多 ObjC 细节

作为长期项目，不建议把大量 Objective-C runtime 细节直接暴露到 Java 侧。

更稳的方式是：

- Java 侧负责生命周期和高层 backend 语义
- native 层负责 Cocoa/Metal 细节
- JNI 边界尽量克制且明确

## 12. MetalExp 的产品与支持边界

项目定位已经明确为：

- 面向少量技术用户的实验性公开项目
- macOS only
- 首发目标是现代 snapshot 线
- 首期不承诺兼容其他渲染 mod

这意味着 issue 支持边界也应当从第一天写清：

- 需要提供诊断日志
- 需要提供系统和 GPU 信息
- 某些 snapshot 漂移问题属于正常维护成本
- 与深度渲染 mod 的冲突首期不作为主要优化目标

## 13. 实施路线建议

推荐分 6 个里程碑推进：

1. bootstrap
2. native surface bring-up
3. resource core
4. shader + pipeline core
5. rendering viability
6. experimental public release

这样做的意义是：

- 每个阶段都能形成真实的可验证进展
- 问题能更早暴露
- 不会长时间困在“感觉架构上可行，但没有用户可见成果”的状态

## 14. 最终评估

### 14.1 是否值得立项

值得。

前提是团队接受：

- 这不是小 mod
- 这不是单纯 mixin 工程
- 这不是几周内自然完工的渲染器

### 14.2 是否适合做成大项目

适合。

原因不是因为它简单，而是因为现代 Blaze3D 架构终于给了足够清晰的 backend seam，让高投入项目有了正确切入点。

### 14.3 最真实的判断

`MetalExp` 是一个合理、昂贵、技术上可信的长期 renderer 项目。

它的核心挑战不会是：

- 如何写更多 mixin

而会是：

- 如何把 backend、shader、native bridge 和 snapshot 适配组织成一个能长期演进的系统

这也正是项目设计 spec 必须足够完整的原因。
